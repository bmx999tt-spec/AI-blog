LINE WORKSに通知する仕組みを作った話

大量の営業メールに埋もれた仕事依頼を見逃さないために。Gmail × Claude AI × LINE WORKS × GitHub Actionsで構築した完全自動通知システムの全記録です。

2026年4月 株式会社Shinker（シンカー） 読了時間：約10分
Gmail API Claude AI LINE WORKS GitHub Actions Python JWT認証 業務自動化

会社のHPお問い合わせフォームから届くメール。その大半は営業メールや自動送信の迷惑メールです。でも、その中には本当の仕事依頼が埋まっています。それを毎回手作業で確認するのは非効率だと感じていました。

そこで「AIにメールを読ませ、仕事依頼だけを社内に自動通知する」仕組みを構築しました。実際に構築してみると、LINE Works APIの認証方式がJWT（Service Account認証）だったり、チャンネルIDの取得方法が独特だったりと、想定外の落とし穴がいくつかありました。この記事ではそのすべてを包み隠さず記録しています。

💡 この記事では構築手順だけでなく、実際にハマったポイントとその解決策も詳しく解説しています。同じ構成を試みる方の参考になれば幸いです。

なぜこのシステムが必要だったか

弊社のinfo@アドレスには、HPからの問い合わせメールが毎日大量に届きます。その内訳はおよそ以下のとおりです。

info@ に届くメールの内訳（概算）

営業・セールスメール
70%
自動送信・メルマガ
25%
✅ 本当の仕事依頼
5%

営業部がこれらをすべて目で確認して選別するのは時間のムダです。しかも見落としのリスクもあります。「仕事依頼だけを自動検出してLINE WORKSの社内全体トークに通知する」ことができれば、営業部は通知が来たときだけ確認して即返信できます。

LINE WORKS API認証の仕組み（JWT方式）

LINE WORKS Bot API v2では、メッセージ送信にService Account認証（JWT方式）が必要です。単純なClient ID＋Secretでは動きません。これが最大の落とし穴でした。

JWT認証フロー

1
JWTトークンを生成 Client ID・Service Account・Private Keyを使ってJWT（署名付きトークン）を作成
↓
2
アクセストークンを取得 LINE Works認証サーバーにJWTを送信してアクセストークンを取得
↓
3
メッセージを送信 アクセストークンを使ってBot APIでトークルームにメッセージ送信

Developer ConsoleでClientAppを作成すると、Service AccountメールアドレスとPrivate Key（.keyファイル）が発行されます。これらをGitHub Secretsに登録して使います。

GitHub Secretsに登録する情報

APIキーなどの秘密情報はコードに直接書かず、すべてGitHub Secretsに保管します。今回登録したSecretsは以下の7つです。

REPOSITORY SECRETS
ANTHROPIC_API_KEYClaude AIのAPIキー

GMAIL_TOKENtoken.jsonの中身

各ファイルが何をしているか

このシステムは5種類のファイルで動いています。それぞれの役割を整理します。

🔑
credentials.json

GoogleへのアクセスIDと秘密鍵が入った証明書ファイルです。Google Cloud ConsoleでOAuth 2.0クライアントIDを作成するとダウンロードできます。「このプログラムはGoogleに登録済みの正規アプリです」という身分証の役割を担います。

🎫
token.json

Gmailへのアクセス許可証です。最初に一度だけauth.pyを実行してGoogleにログインすると自動生成されます。以降はこのファイルで毎回ログイン不要になります。

🐍
auth.py

token.jsonを生成するための初回専用スクリプトです。一度実行すれば二度と使いません。ブラウザを開いてGoogleのOAuth認証を行い、token.jsonを作ります。

⚙️
main.py

システムの本体です。Gmailからメール取得→Claude AIで判定→JWT認証でLINE WORKSトークン取得→通知送信、という一連の処理をすべて担います。

🕐
gmail-checker.yml

GitHub Actionsの設定ファイルです。「毎時0分（UTC）にmain.pyを実行せよ」というスケジュール命令が書かれています。PCの電源が切れていても動き続けます。

構築の手順（全体像）

1
Google Cloud ConsoleでGmail APIを有効化

既存のGoogleアカウントでCloud Consoleにアクセスします。Gmail APIをオンにして、アプリケーションの種類「デスクトップアプリ」でOAuth 2.0クライアントIDを作成し、JSONをダウンロードします。OAuth同意画面のテストユーザーに自分のアドレスを追加することを忘れずに。

2
LINE WORKS Developer ConsoleでClientAppとBot作成

スーパー管理者権限が必要です。ClientAppを作成してOAuth Scopeに「bot」を追加します。次にBotを登録して「複数人のトークルームに招待可」をオンにします。Service AccountのPrivate Keyも発行しておきます。

3
BotをALL業務連絡に招待＆チャンネルIDを取得

Admin画面でBotの公開設定をオンにしてから、LINE WORKSアプリでトークルームにBotを招待します。チャンネルIDはトークルームの右上メニュー→「チャンネルID」から取得できます（数値ではなくUUID形式です）。

4
ローカルPCで初回Gmail認証

auth.pyをPCで1回だけ実行します。ブラウザが開いてGoogleにログインするとtoken.jsonが生成されます。これ以降は不要です。

5
GitHubリポジトリをPrivateで作成

main.pyとgmail-checker.ymlをアップロードします。GitHub Secretsに7つのAPIキー・認証情報を登録して情報漏洩を防ぎます。

6
Actions画面から手動テスト実行

「Run workflow」で動作確認ができます。ログで「通知送信: 201」が表示されれば完成です。以降はUTC毎時0分（日本時間9分）に自動実行されます。

ハマったポイントと解決策

問題1

LINE WORKS Developer Consoleにアクセスできない

解決

スーパー管理者権限が必要でした。一般管理者では入れません。代表に権限付与を依頼することで解決しました。スーパー管理者は複数名に付与できます。

問題2

Gmail OAuth認証で「アクセスをブロック」エラーが出る

解決

Google Cloud ConsoleのOAuth同意画面でテストユーザーに自分のアドレスを追加していませんでした。追加後すぐに解決しました。

問題3

BotをトークルームのBot招待で表示されない

解決

Admin画面のサービス→BotでBotの公開設定が「非公開」になっていました。公開に変更後、表示されるようになりました。

問題4

HPフォームからの問い合わせが「仕事依頼」と判定されない

解決

HPフォームの自動送信メールは送信者がinfo@になるため、AIが「自動送信メール」と誤判定していました。プロンプトに「送信者がinfo@でも本文に問い合わせ内容があればYES」と明示して解決しました。

問題5

LINE WORKS APIが401エラー（認証失敗）

解決

LINE Works API v2はclient_credentials認証に非対応でした。JWT（Service Account認証）に切り替えて解決しました。PyJWTライブラリが必要です。

問題6

LINE WORKS APIが404エラー（送信先が見つからない）

解決

チャンネルIDを数値で設定していましたが、正しくはUUID形式（例：72fbad9e-fe42-87d6-ad6f-99e7194b8493）でした。LINE WORKSアプリのトークルーム設定から「チャンネルID」を取得して解決しました。

まとめ

今回作ったシステムの核心は「AIにメールの意図を判断させる」という部分です。従来のキーワードフィルタリングと違い、文脈を理解して判断するので誤検知が少ないです。

構築にかかった時間は設定作業含めて半日程度。月額費用はほぼゼロ。それでいて営業部の「メールを毎日確認する」という作業が丸ごとなくなりました。

LINE Works APIの認証周りは情報が少なく苦労しましたが、JWT方式とチャンネルIDのUUID形式さえ把握できれば同じ構成で再現できます。

🚀 今後の拡張案：通知メッセージに本文の要約を含める、特定キーワードは即時通知（1時間待たずに）、複数のメールアドレスに対応するなど。
